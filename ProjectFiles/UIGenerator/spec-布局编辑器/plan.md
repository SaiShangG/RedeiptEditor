# 布局编辑器技术方案

## 1. 目标

基于现有需求规范，构建一个使用 `vite`、`react`、`html` 的纯前端布局编辑器首版。首版重点是先让编辑器基础界面、核心编辑流和 JSON 导入导出能力可运行、可验证、可逐步扩展。

该方案仅覆盖首版实现路径，不追求一次性做完全部能力，而是通过阶段性 Demo 逐步交付，每个 Demo 都可以单独运行和验收。

## 2. 技术边界

1. 技术栈限定为 `vite`、`react`、`html`。
2. 首版为纯前端应用，不依赖真实后端。
3. 数据导入方式为粘贴 JSON 文本。
4. 数据导出方式为复制 JSON、下载 JSON 文件。
5. 用户编辑中的布局需要在浏览器本地自动保留。
6. 页面效果以简洁可用为目标，不以高保真视觉设计为目标。
7. 验证方式以手工验收为主，不要求完整自动化测试体系。

## 3. 总体方案

### 3.1 产品形态

首版采用单页应用形态，页面内包含以下核心区域：

1. 树结构区：展示根容器与子对象层级。
2. 画布区：展示当前布局的实时预览。
3. 属性区：展示当前选中对象的可编辑字段。
4. 操作区：提供新建、导入、复制 JSON、下载 JSON 等动作。

### 3.2 数据驱动思路

整个编辑器围绕一份当前布局数据运行。树结构、画布、属性面板、导出内容都基于同一份当前布局数据生成，以保证多视图一致。

### 3.3 首版核心能力

1. 从空白创建布局，并自动生成根容器。
2. 在根容器下添加、删除子对象。
3. 选中对象后同步联动树结构、画布、属性区。
4. 修改对象字段和绑定信息，并实时反映到预览与导出结果。
5. 粘贴 JSON 文本导入已有布局。
6. 复制 JSON 和下载 JSON 文件。
7. 在本地自动保留当前编辑状态。
8. 对重复 ID、空 ID、结构异常等关键问题给出提示并限制无效结果输出。

## 4. 模块划分

### 4.1 应用壳层

负责页面整体布局、区域分栏和主要操作入口。

### 4.2 编辑器状态模块

负责维护当前布局、当前选中对象、校验结果以及本地保留内容，是整页联动的中心。

### 4.3 树结构模块

负责展示对象层级、切换选中对象，并支持基础结构操作。

### 4.4 画布预览模块

负责根据当前布局数据渲染可视化结果，并表达当前选中状态。

### 4.5 属性编辑模块

负责展示并编辑当前选中对象的字段，包括根容器字段、子对象字段和绑定信息。

### 4.6 导入导出模块

负责处理 JSON 文本导入、复制 JSON、下载 JSON 文件。

### 4.7 校验与提示模块

负责识别结构不完整、ID 冲突、非法层级等问题，并将问题反馈给用户。

### 4.8 本地保留模块

负责将当前编辑内容自动保存到浏览器本地，并在再次打开页面时恢复。

## 5. 关键设计决策

### 5.1 纯前端优先

首版不引入后端，目的是尽快验证编辑器的交互模型、数据结构可行性和核心业务流程。

### 5.2 单一数据源

树、画布、属性区、导出 JSON 都基于同一份布局数据，避免多份状态互相漂移。

### 5.3 先保证正确性，再逐步增强体验

首版优先保证对象结构、选中联动、属性编辑、导入导出、基础校验可用，再考虑更复杂的交互体验。

### 5.4 本地自动保留

由于首版没有后端，编辑中断后的恢复能力通过浏览器本地保留实现，降低用户丢失工作内容的风险。

## 6. 分阶段 Demo

## Demo1：构建编辑器基础架构

### Demo 目标

完成应用基础骨架，让页面具备编辑器的主要分区和最小可运行能力。

### Demo 范围

1. 使用 `vite` 创建并运行项目。
2. 搭建页面基础布局，包含树结构区、画布区、属性区、操作区。
3. 页面默认从空白创建一个根容器。
4. 树结构区可以显示根容器。
5. 画布区可以展示根容器的基础预览。
6. 属性区可以展示根容器的基础字段。
7. 用户可以点击根容器并看到明确的选中状态。

### 用户验收

1. 启动页面。
2. 打开后无需额外操作，即可看到完整编辑器界面分区。
3. 在树结构区看到默认根容器。
4. 在画布区看到与根容器对应的预览区域。
5. 点击根容器后，树结构区、画布区、属性区都显示它是当前选中对象。

### 验收通过标准

1. 页面可正常打开。
2. 四个主要区域可见且职责清晰。
3. 默认根容器存在且选中联动成立。

## Demo2：打通结构编辑与属性联动

### Demo 目标

让用户可以真正编辑布局对象，并看到树、属性、画布三者联动。

### Demo 范围

1. 支持在根容器下新增子对象。
2. 支持删除子对象。
3. 支持切换选中不同对象。
4. 支持编辑根容器和子对象的基础字段。
5. 支持编辑子对象的绑定信息。
6. 属性修改后，树结构名称、画布预览、当前布局数据保持一致。
7. 当删除当前选中对象时，页面切换到合理的后续选中状态。
8. 支持基础层级限制，阻止明显非法的位置调整。

### 用户验收

1. 打开页面后新增一个子对象。
2. 在树结构区确认新对象已出现。
3. 点击该子对象，确认画布和属性区同步切换。
4. 修改该对象的显示字段和绑定信息。
5. 观察树结构和画布内容，确认修改立即生效。
6. 删除该子对象，确认树结构和画布同步移除。
7. 再次操作对象层级时，若位置不合法，页面应直接提示不能这样操作。

### 验收通过标准

1. 子对象可新增、选中、编辑、删除。
2. 树、画布、属性区三者联动一致。
3. 非法层级操作会被阻止并提示。

## Demo3：打通 JSON 导入导出、校验与本地保留

### Demo 目标

完成首版闭环，让用户能导入已有配置、修正问题、导出结果并在刷新后恢复内容。

### Demo 范围

1. 支持通过粘贴 JSON 文本导入已有布局。
2. 导入成功后，树结构、画布、属性区正确还原布局内容。
3. 支持复制当前 JSON。
4. 支持下载当前 JSON 文件。
5. 支持本地自动保留当前编辑内容。
6. 页面刷新或重新打开后，自动恢复上次编辑内容。
7. 对重复 ID、空 ID、缺少根容器、多根对象、缺少必需字段等关键问题给出提示。
8. 当当前布局存在关键错误时，不允许导出为有效结果。

### 用户验收

1. 准备一段合法 JSON，并粘贴导入。
2. 导入后确认树结构、画布、属性区内容正确出现。
3. 修改其中一个对象字段。
4. 点击复制 JSON，并检查复制结果是否包含刚才的修改。
5. 点击下载 JSON 文件，并打开文件检查内容是否正确。
6. 刷新页面，确认刚才的编辑内容仍然存在。
7. 再粘贴一段包含重复 ID 或缺少根容器的错误 JSON。
8. 确认页面给出明确错误提示，且不会把该错误配置当作可正常导出的结果。

### 验收通过标准

1. 合法 JSON 可导入并继续编辑。
2. 当前布局可复制和下载为 JSON。
3. 刷新后内容可自动恢复。
4. 关键结构错误可被识别、提示并限制导出。

## 7. 开发顺序建议

1. 先完成最小页面壳层和编辑器主状态。
2. 再接入树结构、画布、属性区的选中联动。
3. 然后补齐新增、删除、编辑等核心操作。
4. 最后接入 JSON 导入导出、本地保留和校验提示。

这样可以保证每一阶段都有稳定可演示结果，而不是到最后才集中联调。

## 8. 风险与应对

### 8.1 多视图状态不一致

风险：树、画布、属性区可能出现显示不同步。

应对：以单一编辑器状态作为唯一数据来源，所有区域只读取和更新这份状态。

### 8.2 导入 JSON 的异常情况较多

风险：已有配置可能不完整、不规范或带有冲突数据。

应对：先覆盖首版明确要求的关键问题类型，并把错误提示做到可理解。

### 8.3 本地保留内容污染当前编辑

风险：历史残留内容可能影响用户重新开始。

应对：提供明确的新建入口，让用户可以从新的默认根容器重新开始。

### 8.4 首版交互复杂度快速膨胀

风险：过早加入复杂拖拽或高级编辑能力，会拖慢首版交付。

应对：首版先聚焦可点击、可编辑、可导入导出、可校验，不强行扩大范围。

## 9. 完成定义

当以下条件全部满足时，可以认为首版技术方案落地完成：

1. 三个 Demo 已全部完成并可独立演示。
2. 用户可从空白开始创建布局。
3. 用户可导入合法 JSON 并继续编辑。
4. 用户可完成对象新增、删除、选中和属性编辑。
5. 用户可复制和下载最终 JSON。
6. 关键结构错误能够被识别并提示。
7. 页面刷新后可自动恢复上次编辑内容。
